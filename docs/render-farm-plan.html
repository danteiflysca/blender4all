A headless LAN render farm for Blender

01 — Overview

System architecture

Three processes, one shared HTTP contract. The coordinator is the only stateful piece: it owns the frame queue, stores the packed .blend, and collects finished frames. Workers are stateless pull-based loops — they ask for work, do it, hand it back. The Blender add-on is just a convenient client for the coordinator's submit endpoint; the farm works without it.

Design decisions locked for v1

DecisionChoiceWhy
Work distributionPull, not pushWorkers request frames when idle. No coordinator-side scheduling, no knowledge of worker capacity, natural load balancing — fast machines simply ask more often.
Unit of workOne frameSimple retry semantics. A dead worker costs you one frame's progress, not a chunk. Frame batching is a v2 optimization.
TransportPlain HTTP, LAN onlyNo WebSockets, no message broker. Everything is a request/response you can replay with curl.
DiscoveryConfig file with coordinator IPmDNS/broadcast discovery is a v2 nicety. You have three machines; type the IP once.
StateSQLite via stdlib sqlite3Survives coordinator restarts (an in-memory dict loses a half-finished 400-frame job to one Ctrl-C). Zero extra dependencies.
AuthSingle shared token, v1One X-Farm-Token header checked by middleware. Not real security — it prevents accidents, not attackers. The farm never leaves the LAN.

Add-on dependency rule

The add-on must use only the Python standard library (urllib.request, json, hashlib). Blender's bundled Python can't be relied on to have pip packages, and vendoring wheels into an add-on is misery you don't need for four HTTP calls.

Ports, paths, conventions

  • Coordinator listens on 0.0.0.0:8420. Workers reach it at http://192.168.x.x:8420 from their config file.
  • Job IDs are uuid4 hex strings. Frame numbers are integers, zero-padded to 4 digits in filenames.
  • Output format for v1 is PNG only (single-file-per-frame keeps the upload endpoint trivial; EXR multilayer works identically but makes files huge — add it in v2 with a format field).
  • All timestamps are UTC ISO-8601. All JSON keys are snake_case.
03 — The Contract

The API contract (build this first, on paper)

Everything else in this plan implements one side of this table. Write it into shared/API.md before any code exists, and when the AI proposes a change mid-build, update the contract file first — it is the single source of truth all three components are built against.

EndpointCallerPurpose
POST /jobsAdd-onMultipart upload: blend_file + JSON params (frame range, step, output format, engine). Returns {job_id}.
GET /jobs/{id}Add-on, status pageFull job status: counts per state, per-frame detail, connected workers.
POST /jobs/{id}/cancelAdd-on, status pageMarks job cancelled; pending frames won't be handed out.
GET /work?worker_id=X&blender_version=YWorkerAtomically claim the next pending frame from any active job. Returns frame assignment or 204 No Content.
GET /jobs/{id}/blendWorkerStreams the packed .blend. Response includes X-Blend-SHA256 header for cache validation.
POST /jobs/{id}/frames/{n}/resultWorkerMultipart upload of the rendered frame, or JSON failure report with the Blender stderr tail.
GET /BrowserStatus dashboard (Phase 4).

Key payloads, verbatim

GET /work → 200 (work available)

{
  "job_id": "a3f8c2...",
  "frame": 147,
  "blend_sha256": "9d41be...",      // worker compares to its cache
  "blend_url": "/jobs/a3f8c2.../blend",
  "output_format": "PNG",
  "engine": "CYCLES",
  "lease_seconds": 1800              // deadline before frame is requeued
}

GET /work → 204 (nothing to do — worker sleeps poll_interval and retries)

POST .../result failure body

{
  "status": "failed",
  "exit_code": 1,
  "stderr_tail": "Error: Out of GPU memory ..."  // last ~4KB, invaluable for debugging
}

Frame state machine

Every frame is a row that moves through exactly these states. This is the heart of the coordinator — get this right and everything else is plumbing.

TransitionTrigger
pending → renderingWorker claims it via GET /work. Row records worker_id, lease_expires_at = now + lease_seconds.
rendering → doneWorker uploads a valid frame file before the lease expires.
rendering → pendingLease expires (worker crashed / lost power / went to lunch). A background task sweeps every 30s. attempts += 1.
rendering → failedWorker reports failure, or attempts reaches 3. Failed frames keep the stderr_tail for the status page.
04 — Build
Phase 1~half a day · target ≤150 lines

Worker agent

Built first, against a stub server, so the trickiest real-world piece — invoking headless Blender correctly with GPU rendering — is proven before any farm exists. The worker is a single file with a single loop and no state beyond a blend cache directory.

Config file — worker/config.toml

coordinator_url = "http://192.168.1.50:8420"
token           = "pick-a-long-random-string"
worker_id       = "shop-pc"          # human-readable, shows on dashboard
blender_path    = "C:/Program Files/Blender Foundation/Blender 4.5/blender.exe"
work_dir        = "C:/farmhand"      # blend cache + render output scratch
poll_interval   = 10                  # seconds to sleep on 204
gpu             = "OPTIX"            # OPTIX | CUDA | HIP | METAL | NONE (CPU)

The loop, in pseudocode

while True:
    work = GET /work?worker_id=...&blender_version=...     # with token header
    if 204: sleep(poll_interval); continue

    if work.blend_sha256 not in local cache:               # cache keyed by hash,
        download blend_url → work_dir/cache/{sha}.blend    # NOT by job id
        verify sha256 of downloaded bytes matches           # corrupt = delete + retry once

    run blender headless on frame N                         # see command below
    if exit 0 and output file exists and size > 0:
        POST result (multipart file upload)
    else:
        POST result {status: failed, exit_code, stderr_tail}
    delete local output file

The Blender invocation — the part worth getting exactly right

blender -b {cache}/{sha}.blend \
  --python-expr "{gpu_setup}" \
  -o "{work_dir}/out/frame_####" \
  -F PNG -noaudio \
  -f {N}

where gpu_setup is this one-liner (built from the config's gpu value; omitted entirely when gpu = "NONE"):

import bpy
prefs = bpy.context.preferences.addons['cycles'].preferences
prefs.compute_device_type = 'OPTIX'
prefs.get_devices()                          # must call this to populate the list
for d in prefs.devices: d.use = True
bpy.context.scene.cycles.device = 'GPU'
Why this is necessary

Headless Blender does not inherit the desktop preferences where you enabled your GPU. Without this expression, every worker silently renders on CPU and you'll wonder why your farm is slower than your workstation. Flag order matters too: -f must come after -o and -F, because Blender processes CLI arguments strictly left to right.

Details the AI will forget unless told

  • Capture stderr and stdout to a rotating buffer; keep the last 4KB for failure reports. Blender's error messages are the only debugging signal you get from a remote machine.
  • Timeout the subprocess at lease_seconds - 60 so a hung Blender doesn't zombie the worker past its lease.
  • Find the actual output file: Blender substitutes #### with the padded frame number — after rendering frame 147 look for frame_0147.png.
  • Network errors never crash the loop. Any request failure logs a line, sleeps, retries. The worker should survive the coordinator rebooting.
  • Report blender --version (parsed once at startup) on every /work call — the coordinator uses it for version-mismatch rejection in Phase 4.
  • Blend cache eviction: keep the 3 most recent blends, delete older ones. Packed files with textures get big.

Acceptance — Phase 1 is done when

  • Against the stub server, the worker downloads a blend, renders the assigned frame, and uploads a valid PNG.
  • GPU is verifiably active during render (watch GPU utilization, or check render time vs. CPU).
  • Kill the stub mid-loop: worker logs the error and keeps polling, no crash.
  • A deliberately broken .blend produces a failed report containing usable stderr.
05 — Build
Phase 2~one day

Coordinator

A FastAPI app whose entire job is bookkeeping: which frames exist, what state they're in, who's working on what. Once this runs, you have a functioning farm you can drive entirely with curl — the add-on is a convenience, not a requirement.

SQLite schema

CREATE TABLE jobs (
  id            TEXT PRIMARY KEY,
  name          TEXT,               -- blend filename, for the dashboard
  blend_sha256  TEXT NOT NULL,
  frame_start   INTEGER, frame_end INTEGER, frame_step INTEGER DEFAULT 1,
  output_format TEXT DEFAULT 'PNG',
  engine        TEXT DEFAULT 'CYCLES',
  status        TEXT DEFAULT 'active',   -- active | complete | cancelled
  created_at    TEXT
);
CREATE TABLE frames (
  job_id           TEXT REFERENCES jobs(id),
  frame            INTEGER,
  state            TEXT DEFAULT 'pending',  -- pending|rendering|done|failed
  worker_id        TEXT,
  lease_expires_at TEXT,
  attempts         INTEGER DEFAULT 0,
  stderr_tail      TEXT,
  render_seconds   REAL,
  PRIMARY KEY (job_id, frame)
);

Endpoint behavior specs

POST /jobs

  • Accepts multipart: blend_file (the packed .blend) and params (JSON string: frame_start, frame_end, frame_step, output_format, engine, name).
  • Streams the upload to storage/blends/{job_id}.blend, computing SHA-256 while writing (packed files can be gigabytes — never buffer in memory).
  • Inserts one frames row per frame in range(start, end + 1, step). Rejects ranges over 10,000 frames with a 422 as a sanity guard.

GET /work — the one piece of genuinely careful code

  • Claiming must be atomic so two workers polling simultaneously never get the same frame. In SQLite, one statement does it:
UPDATE frames SET state='rendering', worker_id=?, lease_expires_at=?
WHERE rowid = (
  SELECT f.rowid FROM frames f JOIN jobs j ON j.id = f.job_id
  WHERE f.state='pending' AND j.status='active'
  ORDER BY j.created_at, f.frame LIMIT 1
) RETURNING job_id, frame;
  • Open SQLite in WAL mode (PRAGMA journal_mode=WAL) and use a single connection guarded by a lock, or set check_same_thread=False with care. This is the concurrency hotspot of the whole system.
  • No pending frames → 204. Also record worker_id → last_seen in a memory dict for the dashboard's worker list.

POST /jobs/{id}/frames/{n}/result

  • Success: stream file to storage/frames/{job_id}/frame_{n:04d}.png, mark done, record render_seconds.
  • Failure: store stderr_tail; requeue as pending if attempts < 3, else mark failed.
  • Reject results for frames this worker doesn't hold (lease expired and reassigned — the late upload is discarded with a 409). When the last frame lands, flip the job to complete.

Lease sweeper

  • An asyncio background task every 30s: any rendering frame with lease_expires_at < now goes back to pending, attempts += 1 (or failed at 3).

Acceptance — Phase 2 is done when

  • The full curl playbook (§09) works end to end with no add-on and no real worker.
  • Two real workers on two machines drain a 20-frame job with zero duplicate renders.
  • Kill a worker mid-frame: within one sweep after lease expiry, the frame is pending again and the other worker picks it up.
  • Restart the coordinator mid-job: state survives, workers reconnect on their next poll, job finishes.
06 — Build
Phase 3~one day

Blender add-on

An N-panel in the 3D viewport sidebar: Render on Farm. It packs, uploads, and shows progress. It never renders anything itself.

Panel layout (Properties → Output-adjacent, or N-panel "Farmhand" tab)

ElementBehavior
Coordinator URL + tokenAdd-on preferences (persisted in Blender prefs, not per-file). Panel shows a red warning row if unset.
Frame rangeDefaults pulled live from scene.frame_start / frame_end / frame_step, overridable.
"Render on Farm" buttonRuns the submit operator below. Disabled while a submission is in flight.
Progress rowdone / total + progress bar factor, plus per-state counts. Appears after submission.
"Cancel job" / "Open output folder"Cancel calls the cancel endpoint; the output button is a v1.1 nicety once frame download exists.

The submit operator, step by step

  1. Guard rails first: file must be saved (bpy.data.is_saved); warn on Blender version mismatch vs. coordinator's known worker versions; warn if any external references remain unpacked after step 2 (missing linked libraries are the classic silent failure).
  2. Pack: bpy.ops.file.pack_all() — embeds textures, but not linked libraries or most cache files (see §08). Then write a temp copy with bpy.ops.wm.save_as_mainfile(filepath=tmp, copy=True) so the user's working file and its saved state are untouched.
  3. Upload the temp copy via multipart urllib request (stdlib multipart is fiddly boilerplate — ~30 lines, write it once in client.py). Delete the temp file after.
  4. Store job_id on the scene (scene.farmhand_job_id string property) so it survives file save/reload.
Do not freeze the UI

Packing + saving a copy is fast, but uploading 2GB is not, and a blocking operator locks all of Blender. Use a modal operator with a timer: kick the upload onto a threading.Thread, and have the modal's TIMER events check a thread-safe flag to report completion and errors. The same pattern drives status polling — a timer fires every 3s, calls GET /jobs/{id} on a thread, and writes results into scene properties, tagging the panel region for redraw. Never call urllib directly from draw().

Acceptance — Phase 3 is done when

  • A textured scene submitted from the panel renders correctly on a worker that has never seen the texture files.
  • Blender stays fully responsive during a large upload.
  • Progress counts update live in the panel while workers churn.
  • Cancel stops new frames from being handed out within one poll cycle.
  • Coordinator unreachable → clean error in the panel, not a console traceback.
07 — Build
Phase 4~two days, spread out

Hardening & status page

Status dashboard — GET /

One static HTML file, vanilla JS, polling GET /jobs/{id} every 3 seconds. No build step, no framework. It shows:

  • Frame grid — one small square per frame, colored by state (exactly the strip at the top of this document). This is the signature view: a glance tells you everything.
  • Worker list — worker_id, Blender version, current frame, last-seen time; stale workers (>60s) dimmed.
  • Failed-frame panel — frame number + stderr tail, with a "requeue" button (POST /jobs/{id}/frames/{n}/requeue, resets attempts).
  • ETA — rolling average of the last 10 render_seconds × remaining frames ÷ active workers. Crude and good enough.

Hardening checklist

  • Version enforcement: coordinator compares the worker's reported major.minor Blender version against the submitting Blender's version (sent in job params); mismatched workers get 204 + a logged warning instead of work.
  • Upload retries: worker retries a failed result upload 3× with exponential backoff (2s/8s/30s) before abandoning — the render already cost minutes; don't waste it on a network blip.
  • Disk guard: coordinator refuses new jobs below 5GB free; worker checks before downloading a blend.
  • Requeue endpoint for failed frames (drives the dashboard button).
  • Frame retrieval: GET /jobs/{id}/frames.zip streaming a zip of finished frames, so pulling results doesn't require filesystem access to the coordinator machine.
  • Run-at-boot docs: a Windows Task Scheduler XML + a systemd unit file for the worker, in the repo README.
08 — Risk

The three genuinely fiddly bits

1 · Blender version parity

Files saved in 4.5 can crash or render differently in 4.2. The system treats major.minor as a hard compatibility key: workers report their version on every poll, the add-on stamps the job with the submitting version, and the coordinator simply refuses to match mismatched pairs. The fix is operational — install the same Blender everywhere — but the farm should tell you rather than produce subtly wrong frames.

2 · Headless GPU configuration

Covered in Phase 1: the --python-expr device setup is mandatory because blender -b ignores your desktop's Cycles device preferences. Per-machine wrinkles: OPTIX needs an NVIDIA driver new enough for the Blender build; a machine whose GPU has less VRAM than the scene needs will OOM — which is exactly what the stderr_tail in failure reports is for. The config's gpu = "NONE" escape hatch keeps a CPU-only machine useful.

3 · Asset packing

pack_all() embeds images and a few other datablocks. It does not pack: linked libraries (other .blend files), simulation caches (fluid/cloth bakes pointing at directories), OS-installed fonts used by text objects, or absolute-path video textures. V1 policy: the add-on checks bpy.data.libraries and known cache users after packing and blocks submission with a clear message listing what won't travel. Make Library Overrides local, bake caches into the file where possible, or don't farm that scene. Detecting and bundling external directories is v2 territory.

09 — Verify

Testing strategy & curl playbook

Stub server (built alongside Phase 1)

stub_server.py: ~50 lines of FastAPI that hard-codes the contract — hands out frames 1–5 of a fixed test .blend, accepts any result, prints what it receives. It exists so the worker is testable before the coordinator does, and stays in the repo forever as the worker's regression harness.

The curl playbook — a full render with no add-on

# 1. Pack a test scene manually in Blender (File → External Data → Pack Resources), save copy

# 2. Submit
curl -X POST http://localhost:8420/jobs \
  -H "X-Farm-Token: $TOKEN" \
  -F "blend_file=@test_scene.blend" \
  -F 'params={"name":"smoke test","frame_start":1,"frame_end":10,"frame_step":1,"engine":"CYCLES","output_format":"PNG"}'
# → {"job_id":"a3f8..."}

# 3. Pretend to be a worker
curl "http://localhost:8420/work?worker_id=fake&blender_version=4.5" -H "X-Farm-Token: $TOKEN"
# → frame assignment JSON. Run again → next frame. State machine is visibly working.

# 4. Upload a fake result
curl -X POST http://localhost:8420/jobs/a3f8.../frames/1/result \
  -H "X-Farm-Token: $TOKEN" -F "frame_file=@any.png"

# 5. Watch status
curl http://localhost:8420/jobs/a3f8... -H "X-Farm-Token: $TOKEN"

Failure drills to run before calling it done

  1. Power test: pull the plug (or kill -9) on a worker mid-render. Frame requeues after lease expiry; job completes.
  2. Restart test: restart the coordinator mid-job. Nothing is lost; workers resume within one poll.
  3. Bad scene test: submit a .blend referencing a missing linked library and confirm the add-on blocks it at submit time.
  4. Slow network test: throttle a worker's connection; confirm result-upload retries succeed and no frame is double-rendered.
  5. The real test: a genuine 200+ frame animation across all machines overnight, then scrub the sequence for any frame rendered with wrong settings (the CPU-fallback frames render fine — they're just suspiciously slow and sometimes noisier).
10 — Process

Working with AI on this build

This plan is structured so each phase is one self-contained Claude Code session with a testable exit condition. Practices that keep the build on rails:

  • Contract first, always. Start the very first session by having it write shared/API.md from §03 of this document. Every later session gets that file in context. When behavior needs to change, edit the contract, then the code.
  • One phase per session. Give it the phase section plus the contract, not the whole plan. Smaller context, sharper output.
  • Paste the acceptance checklist as the definition of done and have it write the stub-server test or curl script that proves each item — the checklists in this doc are written to be machine-verifiable on purpose.
  • Feed it the gotchas verbatim. The GPU python-expr, the CLI argument ordering, the pack_all() blind spots, and the atomic-claim SQL are exactly the details a model will get subtly wrong from memory. They're spelled out in full above so you can paste rather than trust.
  • Keep the worker to one file. Resist AI's instinct to abstract it into modules — a 150-line script you can read top to bottom is the feature.
  • Commit at every green checklist. Each phase boundary is a tagged, working farm.

Suggested session order

  1. Repo scaffold + shared/API.md + stub server
  2. Worker agent against the stub (test on your workstation before touching helper PCs)
  3. Coordinator + curl playbook, then swap the worker's config from stub to real
  4. Add-on: client.py + submit operator first, panel polish second
  5. Dashboard + hardening items, one checklist row at a time
11 — Scope

Explicitly out of scope for v1

Written down so neither you nor the AI drifts into building them early:

  • Auto-discovery (mDNS/UDP broadcast) — config file IPs are fine for three machines.
  • Frame chunking / tile splitting — one frame per assignment until per-frame overhead is measurably the bottleneck.
  • EXR / multilayer / video output — PNG only; the format field exists in the schema so this is additive later.
  • Worker as a Tauri/desktop app — the daemon plus the web dashboard covers observability.
  • Multi-user auth, HTTPS, internet exposure — this farm lives and dies on the LAN.
  • Linked-library bundling — v1 detects and refuses; v2 can collect and rewrite paths.
  • Priority queues / multiple concurrent jobs racing — jobs are FIFO by creation time; that's the ORDER BY in the claim query and it's enough.
